# 공통 코드 스타일 규칙

## 한 문장 정의

이 문서는 언어와 무관하게 모든 프로젝트에서 지켜야 하는 코드 작성·주석·에러 처리의 최소 기준입니다.

## 무엇을 하는가

언어별 포매터(Black, Prettier 등) 설정이나 린터 규칙은 `rules/languages/<lang>/README.md`에 둡니다. 이 문서는 **언어와 무관한** 판단 기준만 담습니다. 팀 선호의 문제(들여쓰기 폭, 따옴표)는 여기 두지 않고, "왜 중요한가"가 언어를 넘어 동일한 항목만 둡니다.

## 왜 필요한가

AI 에이전트는 세션마다 같은 프로젝트에서 일관된 코드를 써야 합니다. 포매터가 해결하지 못하는 판단(이름의 서술성, 함수 크기, 에러의 구체성, 주석의 목적)은 사람이 합의해 기록해둬야 에이전트가 반복 결정할 수 있습니다.

## 기본 원칙

### 1. 네이밍

- 프로젝트가 선택한 표기법(snake_case / camelCase)을 **파일 안에서 섞지 않습니다**.
- 이름은 "무엇을 하는가"보다 "무엇인가"를 우선합니다. `get_user()`보다 `user(id)` 처럼.
- 한 글자 변수는 루프 인덱스(`i`, `j`) 또는 수학 컨벤션(`x`, `y`, `n`)에서만.
- 불리언 이름은 긍정문. `is_ready`, `has_permission`. `not_disabled` 금지.
- 약어는 업계 표준(URL, HTTP, JSON, DB)만. 팀 내부 약어는 풀어 씁니다.

### 2. 함수·메서드 크기

- 한 함수는 한 가지 일만 합니다. 동사가 여러 개면 나눕니다.
- 긴 함수(대략 40줄 이상)는 분해를 **검토**합니다. 분해가 이해를 해치면 유지합니다(인위적 쪼개기 금지).
- 매개변수 4개를 넘으면 구조체/딕셔너리/옵션 객체로 묶는 것을 검토합니다.
- 중첩 깊이 3단을 넘으면 조기 반환(early return) 또는 별도 함수로 평탄화합니다.

### 3. 주석

- 코드는 "무엇을 하는지"를 보여주고, 주석은 "왜 그렇게 하는지"를 적습니다.
- 자명한 주석(`i += 1  # i를 1 증가`) 금지.
- `TODO:`는 **소유자와 조건**을 포함합니다. `TODO(name): refactor when <condition>`.
- `FIXME:`는 **원인과 임시방편**을 함께 적습니다. `FIXME: retries leak sockets; cap to 1 for now`.
- `XXX:`는 설명 없이 쓰지 않습니다. 이유가 없다면 제거합니다.
- 주석이 코드와 모순되면 **주석을 믿지 않고 코드를 먼저 검증**합니다.

### 4. 에러 처리

- 조용한 실패 금지. 예외를 먹어서 반환값 `None`으로 섞지 않습니다.
- 예외는 가장 구체적인 타입을 잡습니다. `except Exception:` / `catch (e)` 전반 처리는 재발생(`raise` / `throw`) 없이 쓰지 않습니다.
- 에러 메시지는 **무엇이 실패했고, 어떤 값으로, 사용자가 뭘 확인해야 하는지**를 포함합니다. 스택트레이스 그대로 사용자에게 노출하지 않습니다(디버그 모드 제외).
- 복구 불가능한 상태에서는 프로세스를 종료하거나 명시적으로 블록을 통과시킵니다. 불투명한 계속 실행 금지.

### 5. 매직 넘버·문자열

- 리터럴 상수는 상수 변수로 이름을 줍니다. `30`보다 `MAX_RETRIES = 30`.
- 테스트 픽스처와 설정 파일은 예외입니다(값 자체가 문서 역할).

### 6. 추상화 규칙

- 중복이 **세 번** 나타날 때만 추상화합니다 (rule of three).
- 한 번 쓰는 추상화, 구현이 하나뿐인 인터페이스는 만들지 않습니다.
- 다형성·상속보다 composition을 먼저 시도합니다.

### 7. 의존성·import

- 외부 모듈 → 내부 모듈 순서.
- 순환 의존 금지. 발견하면 공통 베이스를 분리합니다.
- 미사용 import는 제거합니다(에디터/린터 자동 처리해도 됨).
- 전역 상태 추가 전에 매개변수로 전달할 수 있는지 검토합니다.

### 8. 커밋·변경 메시지

- 제목은 명령형 동사로 시작(`Add`, `Fix`, `Refactor`, `Remove`).
- 72자 이내 제목. 본문에 "왜"를 적습니다.
- 한 커밋 = 한 논리적 변경. 포맷 변경과 동작 변경을 섞지 않습니다.
- 커밋 메시지 끝에 붙이는 공동저자/봇 표시는 팀 컨벤션을 따릅니다.

### 9. 죽은 코드·디버그 잔재

- 주석으로 남긴 구버전 코드는 PR에서 제거합니다. 필요하면 git 히스토리에서 찾습니다.
- `console.log` / `print` / `dump` 등 디버그 출력은 머지 전에 지웁니다.
- 조건부 `if False:` 블록, 사용 안 되는 분기는 제거합니다.

### 10. 변수 범위

- 가장 좁은 스코프에서 선언합니다. 블록 상단에 몰아넣지 않습니다(언어가 허용하면).
- 가변 전역은 피합니다. 상수 전역은 허용.
- 큰 반환값을 변수에 담아 그대로 반환하는 불필요한 중간 변수는 제거합니다.

## 결과는 무엇인가

이 원칙을 따르면 에이전트의 출력이 기존 코드와 같은 결로 섞입니다. 코드 리뷰에서 스타일 지적이 줄고, "왜 이렇게 썼나"를 쫓는 시간이 줄어듭니다.

## 기능 추가/수정 판단 기준

새 항목은 **언어를 넘어서 동일하게 의미 있을 때만** 추가합니다. 특정 언어에만 해당하면 `rules/languages/<lang>/`로 보냅니다. 팀 개별 선호(들여쓰기 폭, 따옴표 종류)는 이 문서에 두지 않습니다.
