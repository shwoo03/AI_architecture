# MCP 디스플린 규칙

## 한 문장 정의

MCP 디스플린은 프로젝트에 연결한 MCP 서버와 도구 수를 절제해 컨텍스트 토큰 예산을 보호하는 규칙입니다.

## 무엇을 하는가

각 MCP 도구는 이름, 설명, 파라미터 스키마가 매 대화 시작 시점에 컨텍스트에 포함됩니다. 도구가 많아질수록 실제 작업에 쓸 수 있는 토큰 여유가 줄어듭니다. 이 규칙은 프로젝트당 상한을 정해 토큰을 보전합니다.

## 왜 필요한가

ECC (everything-claude-code) 사례에 따르면 MCP 도구가 과도해지면 200K 컨텍스트 윈도우가 실사용 기준 ~70K까지 줄어듭니다. 도구 하나하나가 유용해 보여도 누적되면 긴 세션에서 모델이 조기에 컨텍스트 한계에 도달합니다. 사전 절제가 사후 압축보다 저렴합니다.

## 기준

- **프로젝트당 활성 MCP 서버: ≤ 10개**
- **활성 도구 총합: ≤ 80개** (모든 MCP 서버의 도구를 합산)
- 새 MCP를 추가할 때마다 "이 서버의 도구 중 실제로 쓰는 것이 몇 개인가?"를 확인합니다. 불필요한 도구는 비활성화합니다.

## 상한을 초과해야 할 때

특정 프로젝트가 기준을 넘어야 한다면 **예외를 명시적으로 기록**합니다.

- `docs/PROJECT_OPERATING_PLAN.md`에 초과 사유(도구별 실사용 빈도, 대체 불가능성)를 기록합니다.
- 가능하면 세션 시작 시 `/mcp` 또는 설정에서 불필요한 도구를 숨깁니다.

## 어떻게 확인하는가

- Claude Code: `/mcp` 명령으로 활성 서버와 도구 수 확인.
- 다른 하네스: 각 도구의 프로필/설정에서 목록 확인.
- 새 서버를 붙일 때마다 누적 도구 수를 다시 셉니다.

## 결과는 무엇인가

이 규칙을 지키면 다음 상태를 유지합니다.

- 장시간 세션에서 컨텍스트 여유가 더 오래 지속됩니다.
- 에이전트가 모든 도구 설명을 읽지 않아도 관련 도구를 빠르게 고릅니다.
- 조기 auto-compact가 줄어들어 작업 맥락이 더 오래 남습니다.

## 기능 추가/수정 판단 기준

기준 숫자(10/80)를 바꿀 때는 실제 토큰 측정 근거와 함께 변경합니다. 숫자 완화는 토큰 여유가 입증되었을 때만, 강화는 여러 프로젝트에서 공통으로 부담이 확인됐을 때만 합니다.
