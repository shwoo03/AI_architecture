# 공용 AI 프로젝트 운영 시스템

## 한 문장 정의

이 저장소는 새 AI 프로젝트를 시작할 때마다 같은 방식으로 목표를 정리하고, 에이전트를 투입하고, 검증하고, 기록하고, 다음 결정을 내리게 만드는 공용 운영 시스템입니다.

## 무엇을 하는가

이 시스템은 “AI에게 일을 시킨다”를 반복 가능한 운영 절차로 바꿉니다. 프로젝트 목표를 먼저 얇게 정리하고, 필요한 작업 흐름과 에이전트 역할을 고른 뒤, 작은 검증 단위로 실행 결과를 확인합니다. 실행 중 생긴 판단과 증거는 로그와 문서에 남겨 다음 세션이나 다른 에이전트가 같은 맥락을 이어받을 수 있게 합니다.

주요 기능은 다음과 같습니다.

- 프로젝트 목표, 성공 기준, 제약 조건을 먼저 고정합니다.
- Codex와 Claude가 같은 운영 원칙을 읽도록 합니다.
- 반복되는 업무를 워크플로와 에이전트 역할로 분리합니다.
- 작업 결과를 검증 가능한 증거로 남깁니다.
- 장기 지식은 출처가 있는 항목으로만 축적합니다.
- Notion 문서는 사용자가 기능 추가와 수정을 결정할 수 있도록 설명 중심으로 유지합니다.

## 왜 필요한가

AI 프로젝트가 커지면 가장 먼저 무너지는 부분은 코드가 아니라 맥락입니다. 어떤 목표 때문에 이 기능이 필요한지, 어떤 기준으로 성공을 판단하는지, 어떤 에이전트가 어느 범위까지 책임지는지 흐려지면 세션이 길어질수록 결정 품질이 떨어집니다.

이 시스템은 그 문제를 막기 위해 다음 질문에 항상 답하게 만듭니다.

- 지금 만들려는 것은 무엇인가?
- 왜 지금 필요한가?
- 어떻게 동작해야 하는가?
- 결과가 좋다는 것은 무엇으로 판단하는가?
- 기능을 추가해야 하는가, 기존 기능을 수정해야 하는가, 보류해야 하는가?

## 어떻게 동작하는가

새 프로젝트는 먼저 프로젝트 프로필을 채웁니다. 프로필은 프로젝트의 목표, 사용자, 성공 기준, 실패 기준, 제약 조건, 사용할 기술 스택, 활성화할 워크플로를 담는 얇은 계약입니다.

그다음 에이전트는 운영 루프를 따릅니다.

1. 프로젝트 목표와 성공 기준을 확인합니다.
2. 필요한 경우 명세와 검증 기준을 만듭니다.
3. 가장 작은 실행 단위를 계획합니다.
4. 필요한 에이전트 역할과 권한을 선택합니다.
5. 구현하거나 조사합니다.
6. 가장 싼 검증부터 실행합니다.
7. 결과를 기록하고 계속할지, 수정할지, 재설계할지, 중단할지 결정합니다 (`docs/OPERATING_LOOP.md` "6. 결정" 참고).

## 결과는 무엇인가

이 시스템을 사용하면 새 프로젝트마다 다음 결과물이 생깁니다.

- 사용자가 이해할 수 있는 프로젝트 목표와 성공 기준
- 에이전트가 실행할 수 있는 작업 흐름
- 반복 업무에 재사용할 수 있는 스킬과 규칙
- 검증 결과와 실행 로그
- 다음 세션이 바로 이어받을 수 있는 인수인계 기록
- Notion에서 읽기 쉬운 기능 중심 문서

## 기능 추가/수정 판단 기준

기능을 추가하거나 수정할 때는 먼저 `docs/FEATURE_DECISION_GUIDE.md`의 기준을 따릅니다. 핵심 판단은 다음과 같습니다.

- 새 기능으로 추가: 기존 구조가 해결하지 않는 사용자 목표가 있고, 성공 기준을 독립적으로 검증할 수 있을 때.
- 기존 기능 수정: 목적은 맞지만 설명, 입력, 출력, 검증 방식이 부족해 사용자가 결정을 못 할 때.
- 보류: 목표나 성공 기준이 불명확하거나 검증 방법이 아직 없을 때.
- 삭제 또는 축소: 같은 역할을 하는 기능이 이미 있거나 유지 비용이 가치보다 클 때.

## 시작 방법

권장 방식은 부트스트랩 스크립트를 사용하는 것입니다. 이 방식은 공용 시스템 내부 상태를 새 프로젝트에 잘못 복사하지 않도록 정리합니다.

```powershell
python scripts/bootstrap/new-project.py --name <name> --target <path> --domain <domain> --stack <stack>
```

부트스트랩 직후 순서는 다음과 같습니다.

1. 생성된 프로젝트 디렉터리로 이동합니다.
2. `python scripts/verify-skeleton.py`로 복사 상태를 검증합니다. 종료 코드 0이면 준비 완료입니다.
3. Claude Code는 `CLAUDE.md`를, Codex는 `AGENTS.md`를 진입 문서로 씁니다. 에이전트가 한두 가지씩 질문해 `docs/PROJECT_PROFILE.md`를 함께 채웁니다.
4. 첫 프롬프트 예시: "이 프로젝트는 <목표> 해. PROJECT_PROFILE 같이 채워줘."

모르는 사실은 추측하지 않고 `[NEEDS CLARIFICATION: <질문>]` 또는 `TBD`로 명시적으로 보류합니다. 표시 규칙의 canonical 정의는 `rules/common/README.md`의 "모르는 사실 표시 규칙" 섹션입니다.

### Notion은 선택

`docs/NOTION_DOCUMENTATION_RULES.md`와 관련된 규칙은 프로젝트가 실제로 Notion 워크스페이스를 쓸 때만 적용합니다. Notion을 쓰지 않는 프로젝트는 해당 규칙과 `AGENTS.md`의 Notion 관련 항목을 무시해도 됩니다.

## 문서 작성 기준

모든 설명 문서는 한국어를 기본으로 합니다. 경로나 파일 목록은 문서의 주 설명 방식으로 쓰지 않습니다. 필요한 경우 마지막의 “구현 연결 정보”에만 보조 정보로 둡니다.

각 문서는 가능한 한 다음 질문에 답해야 합니다.

- 무엇을 하는가?
- 왜 필요한가?
- 어떻게 동작하는가?
- 결과는 무엇인가?
- 기능을 추가하거나 수정할 때 무엇을 봐야 하는가?
