# 에이전트 권한 기준

## 한 문장 정의

이 문서는 반복 에이전트가 어디까지 읽고, 어디까지 쓰고, 언제 제안만 해야 하는지 정하는 기본 권한 기준입니다.

## 무엇을 하는가

역할별 기본 경계를 정합니다. 전략 역할은 주로 계획과 제안을 만들고, 탐색 역할은 읽기만 하며, 구현 역할은 지정된 범위만 수정하고, 검증 역할은 결과 확인과 검증 기록에 집중합니다.

## 왜 필요한가

에이전트가 권한 없이 핵심 문서나 지식을 바꾸면 시스템 신뢰도가 떨어집니다. 반대로 모든 일을 승인 대상으로 만들면 작업 속도가 지나치게 느려집니다. 이 문서는 역할별 안전한 기본값을 정해 둘 사이의 균형을 잡습니다.

## 기본 경계

| 역할 | 읽기 | 쓰기 | 쓰기 대신 제안해야 하는 것 |
| --- | --- | --- | --- |
| 전략 | 운영 문서, 지식, 런타임 기록 | 제안 영역 | 핵심 아키텍처, 거버넌스, 장기 지식 |
| 탐색 | 배정된 자료 | 없음 | 모든 수정 |
| 구현 | 배정된 구현 범위 | 배정된 구현 범위 | 범위 확장, 규칙, 거버넌스 |
| 검증 | 프로젝트 파일과 런타임 로그 | 검증 결과와 제안 | 소스 변경, 장기 지식 |

## 어떻게 동작하는가

구현 역할이 파일을 수정하기 전에는 다음이 정해져야 합니다.

- 소유 범위
- 명시적으로 제외되는 범위
- 기대 산출물
- 검증 명령 또는 검토 방식
- 되돌림이나 롤백 기대치

## 기본 차단 규칙

에이전트는 직접 다음을 수행하지 않습니다.

- 런타임 소유 설정 변경
- 승인 또는 샌드박스 설정 변경
- 로그, 검증 증거, 지식 항목 삭제
- 장기 지식 직접 수정
- 승인 없는 제안 적용
- 자기 권한 확장

## 결과는 무엇인가

이 기준을 적용하면 에이전트 작업은 다음 상태가 됩니다.

- 쓰기 범위가 작업 전에 명확합니다.
- 권한 초과는 제안이나 사용자 확인으로 전환됩니다.
- 핵심 문서와 장기 지식은 사람 승인 없이 바뀌지 않습니다.
- 검증과 구현의 책임이 분리됩니다.

## 기능 추가/수정 판단 기준

권한을 추가하거나 넓힐 때는 먼저 줄이려는 위험과 검증 방법을 설명해야 합니다. 반복 사용되지 않는 일회성 권한은 문서에 고정하지 않습니다. 에이전트가 직접 적용할 필요가 없는 변경은 제안으로 남기는 것이 기본입니다.

## 승인 기록 예시

```json
  "phase": "governance",
  "action": "permission_decision",
  "project": "<project-name>",
  "data": {

    "scope": [],
    "reason": ""
  }
}
```
