# Codex 공통 규칙 안내

## 한 문장 정의

이 문서는 Codex 전용 규칙 문서가 어떤 역할을 하고, 언제 프로젝트별 규칙으로 확장해야 하는지 설명합니다.

## 무엇을 하는가

공통 규칙은 프로젝트를 가리지 않고 적용되는 가드레일입니다. 범위, 안전, 품질, 기록, 자동화 경계를 짧고 명확하게 둡니다. 자세한 절차는 스킬이나 워크플로에 둡니다.

## 왜 필요한가

규칙과 절차가 섞이면 어떤 내용이 반드시 지켜야 할 조건인지, 어떤 내용이 작업 방법인지 구분하기 어렵습니다. 이 문서는 Codex 규칙 영역을 짧은 기준으로 유지하기 위해 필요합니다.

## 어떻게 동작하는가

규칙은 다음 범주로 나눕니다.

- 범위: 프로젝트에 포함되는 것과 제외되는 것
- 안전: 파괴적 작업, 비밀 정보, 데이터 노출, 승인 조건
- 품질: 테스트, 검증, 리뷰, 수용 기준
- 기록: 활동 로그에 남겨야 하는 것
- 자동화: 자동 제안과 사람 승인이 필요한 것

## 기본 규칙

- 사용자 작업이나 관련 없는 변경을 덮어쓰지 않습니다.
- 프로젝트 프로필의 성공 기준을 기준으로 삼습니다.
- 패턴을 승격하기 전에 검증합니다.
- 로그는 append-only로 다룹니다.
- 생성된 변경은 승인 전까지 제안으로 둡니다.
- 피벗 신호가 발생하면 방향을 바꾸기 전에 증거를 기록합니다.

## 결과는 무엇인가

Codex 규칙을 이렇게 유지하면 핵심 가드레일은 작고 명확하게 남고, 긴 절차와 예시는 필요한 때만 로드되는 스킬로 분리됩니다.

## 기능 추가/수정 판단 기준

새 규칙은 모든 프로젝트에 반복적으로 적용되는 불변 조건일 때만 추가합니다. 절차, 체크리스트, 예시는 스킬이나 워크플로로 옮깁니다. 프로젝트 하나에만 해당하는 예외는 프로젝트 프로필이나 운영 계획에 둡니다.

## 체크리스트 템플릿

- [ ] 목표와 비목표가 명확합니다.
- [ ] 현재 작업이 활성 프로젝트 프로필과 연결됩니다.
- [ ] 쓰기 범위가 명시되어 있습니다.
- [ ] 검증 방법이 알려져 있습니다.
- [ ] 주요 결정은 활동 로그에 남았습니다.
- [ ] 검증 실패 후 피벗 기준을 확인했습니다.
