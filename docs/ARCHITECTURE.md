# 공용 AI 시스템 아키텍처

## 한 문장 정의

이 아키텍처는 프로젝트 목표를 에이전트 실행, 검증, 기록, 지식 축적으로 연결하는 공용 운영 구조입니다.

## 무엇을 하는가

이 시스템은 새 프로젝트가 생겼을 때 매번 처음부터 운영 방식을 다시 만들지 않도록 공통 구조를 제공합니다. 목표 정의, 규칙 적용, 스킬 선택, 에이전트 역할 분리, 실행 기록, 검증, 지식화, 피벗 판단을 하나의 흐름으로 묶습니다.

핵심 역할은 다음과 같습니다.

- 프로젝트별 사실과 공통 방법론을 분리합니다.
- 에이전트가 읽어야 할 정보를 순서화합니다.
- 반복 작업을 워크플로와 에이전트 역할로 관리합니다.
- 검증 전에는 장기 지식으로 확정하지 않습니다.
- 자동화는 승인 전까지 제안으로 남깁니다.

## 왜 필요한가

AI 프로젝트는 대화가 길어질수록 “목표”, “검증 기준”, “누가 무엇을 책임지는지”가 흐려집니다. 이 상태에서 기능을 추가하면 시스템은 커지지만 사용자가 판단하기는 더 어려워집니다.

아키텍처는 다음 문제를 막기 위해 존재합니다.

- 프로젝트별 정보가 공통 규칙에 섞이는 문제
- 에이전트가 권한이나 목표 없이 작업하는 문제
- 검증되지 않은 주장이 지식으로 쌓이는 문제
- 실패한 작업이 기록되지 않아 같은 실수를 반복하는 문제
- Notion과 로컬 문서가 서로 다른 구조로 설명되는 문제

## 어떻게 동작하는가

시스템은 세 층으로 나뉩니다.

- 범용 기반: 모든 프로젝트에 적용되는 규칙, 운영 루프, 권한, 검증 기준입니다.
- 도메인 스킬: 특정 반복 업무가 있을 때만 활성화되는 절차와 체크리스트입니다.
- 프로젝트 오버레이: 현재 프로젝트의 목표, 사용자, 성공 기준, 제약 조건입니다.

실행 흐름은 단순합니다.

1. 프로젝트 목표와 성공 기준을 확인합니다.
2. 공통 규칙과 운영 루프를 적용합니다.
3. 필요한 스킬이나 워크플로만 선택합니다.
4. 작업이 크거나 반복되면 에이전트 역할을 나눕니다.
5. 실행 결과를 로그로 남깁니다.
6. 가장 작은 검증부터 수행합니다.
7. 검증된 내용만 지식이나 문서 개선으로 제안합니다.

## 결과는 무엇인가

이 아키텍처를 따르면 다음 결과를 얻습니다.

- 새 프로젝트를 빠르게 시작할 수 있습니다.
- 에이전트가 목표와 권한을 잃지 않습니다.
- 반복 업무를 역할과 워크플로로 재사용할 수 있습니다.
- 검증 결과와 활동 기록이 남습니다.
- 기능 추가와 수정 판단이 문서화됩니다.
- 장기 지식은 출처와 함께 관리됩니다.

## 기능 추가/수정 판단 기준

아키텍처를 바꿔야 하는지 판단할 때는 다음 기준을 사용합니다.

- 새 책임이 모든 프로젝트에 반복적으로 필요하면 범용 기반에 추가합니다.
- 특정 도메인이나 작업 방식에만 필요하면 스킬로 둡니다.
- 한 프로젝트에만 필요한 사실이면 프로젝트 오버레이에 둡니다.
- 설명 부족으로 사용자가 결정을 못 한다면 새 기능보다 문서 수정이 우선입니다.
- 자동화가 핵심 문서를 직접 바꾸려 한다면 제안 단계로 낮춥니다.

## 안전 경계

- 핵심 규칙과 아키텍처 문서는 사람이 승인해야 바꿉니다.
- 자동 생성 내용은 먼저 제안 영역에 둡니다.
- 활동 로그와 에이전트 실행 로그는 append-only로 다룹니다.
- 지식 문서는 에이전트가 직접 수정하지 않습니다.
- 프로젝트가 중단되면 실패 이유와 재사용 가능한 교훈을 먼저 남깁니다.
