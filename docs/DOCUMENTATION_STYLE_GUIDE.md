# 문서화 스타일 가이드

## 한 문장 정의

이 문서는 로컬 문서와 Notion 문서를 모두 “사용자가 기능을 이해하고 결정할 수 있는 설명”으로 유지하기 위한 작성 기준입니다.

## 무엇을 하는가

문서가 경로 목록, 속성 표, 내부 구조 설명에 머무르지 않도록 공통 설명 형식을 정합니다. 모든 주요 문서는 사용자가 다음을 판단할 수 있어야 합니다.

- 이 기능이 무엇인지
- 왜 필요한지
- 어떻게 동작하는지
- 결과가 무엇인지
- 기능을 추가하거나 수정하려면 무엇을 봐야 하는지

## 왜 필요한가

경로나 파일 목록 중심 문서는 에이전트에게는 유용할 수 있지만, 사용자가 제품이나 시스템 개선 결정을 내리는 데는 부족합니다. “어떤 역할을 하는지”, “왜 있는지”, “어떻게 판단해야 하는지”가 없으면 문서는 읽어도 행동으로 이어지지 않습니다.

이 기준은 문서를 개발자용 내부 지도에서 의사결정용 설명서로 바꾸기 위해 필요합니다.

## 독자 기준

문서의 기본 독자는 해당 기능을 처음 보는 사람입니다. 독자가 이전 세션, 코드 변경 내역, 파일 경로,
대화 맥락을 알고 있다고 가정하지 않습니다. 좋은 문서는 독자가 한 페이지를 읽고 기능의 역할, 만든
이유, 동작 순서, 결과물, 실패 신호, 다음 변경 기준을 설명할 수 있게 만듭니다.

## 어떻게 동작하는가

문서는 **필수 7-섹션 (Core)** 과 **선택 3-섹션 (Extended)** 로 구성합니다.

### 필수 7-섹션 (모든 주요 문서에 적용)

1. 한 문장 정의
2. 무엇을 하는가 (또는 무엇을 해결하는가)
3. 왜 필요한가
4. 사용 시점
5. 어떻게 동작하는가
6. 결과는 무엇인가
7. 기능 추가/수정 판단 기준
8. 구현 연결 정보

현재 공용 시스템의 핵심 문서(`AGENTS.md`, `CLAUDE.md`, `README.md`, `docs/ARCHITECTURE.md`,
`docs/OPERATING_LOOP.md`, `docs/SESSION_CONTINUITY.md` 등)가 이 구조를 사용합니다.

### 선택 확장 섹션 (판단이 핵심인 문서에서만 추가)

사용자가 기능 추가/수정을 결정해야 하는 페이지에는 다음을 추가합니다.

- 사용자가 겪는 문제
- 입력과 출력
- 실패 신호

이 세 섹션은 "결과는 무엇인가" 앞뒤에 배치하거나 "기능 추가/수정 판단 기준" 바로 앞에
배치합니다. Notion의 대표 페이지, 기능 가이드, 결정 가이드처럼 의사결정에 바로 연결되는
문서에서 주로 사용합니다.

모든 섹션을 억지로 길게 채울 필요는 없습니다. 필수 7개는 반드시, 확장 3개는 결정 맥락이
있을 때만.

## 결과는 무엇인가

이 기준을 적용한 문서는 다음 상태가 됩니다.

- 사용자가 먼저 읽는 내용은 경로가 아니라 역할과 판단 기준입니다.
- 표는 비교나 상태 요약에만 사용됩니다.
- 경로와 파일명은 마지막 보조 정보로 이동합니다.
- 로컬 문서와 Notion 문서가 같은 질문에 답합니다.
- 다음 에이전트가 문서 품질을 점검할 수 있습니다.

## 작성 규칙

- 한국어를 기본 언어로 사용합니다.
- 파일 경로나 디렉터리 경로를 설명의 중심에 두지 않습니다.
- 경로가 필요하면 “구현 연결 정보” 섹션에만 둡니다.
- 기능 문서의 제목과 첫 섹션은 날짜, 파일, 작업 로그가 아니라 기능 또는 결정 주제를 앞세웁니다.
- 처음 보는 사람이 이해할 수 있도록 역할, 만든 이유, 동작 흐름, 사용 시점, 결과물, 실패 신호를 풀어서 씁니다.
- 표는 결정 비교, 상태 요약, 체크리스트에만 사용합니다.
- 표의 첫 열을 경로나 파일명으로 시작하지 않습니다.
- “무엇/왜/어떻게/결과/판단 기준” 중 하나라도 빠지면 문서가 의사결정에 부족하다고 봅니다.
- 핵심 기능 문서는 성공 상태와 실패 신호를 함께 설명합니다.
- 하위 항목 문서는 부모 기능 안에서의 역할과 언제 읽어야 하는지를 먼저 설명합니다.
- 도메인 예시는 보조 설명으로만 사용하고, 핵심 설명은 범용 프로젝트에 적용 가능해야 합니다.
- 추측한 사실은 확정 표현으로 쓰지 않습니다.
- 장기 지식 항목은 반드시 출처 포인터를 포함합니다.

## Notion 작성 규칙

Notion에서는 데이터베이스 페이지 하나를 문서 하나로 취급합니다. 페이지 안에 자식 페이지를 만들지 않습니다. 관계, 상태, 계층 같은 구조 정보는 데이터베이스 속성으로 표현하고, 본문은 사용자가 읽고 판단할 수 있는 설명을 담습니다.

사용자에게 기본으로 제공하는 Notion 보기는 `기능별 문서` 하나로 둡니다. 이 보기는 경로나 내부 relation 컬럼을 앞세우지 않고, 이름, 계층, 문서 유형, 상태, 기능 요약만 보여줍니다. 여러 보기를 만들면 사용자는 어떤 보기가 최신 기준인지 판단하기 어렵기 때문에, 추가 보기는 만들지 않는 것을 기본으로 합니다.

Notion 페이지의 첫 화면에는 다음 정보가 보여야 합니다.

- 이 문서가 다루는 기능
- 왜 읽어야 하는지
- 사용자가 내릴 수 있는 결정
- 다음에 읽을 문서 또는 실행할 행동

Notion 페이지가 당일 구현 기록이라도 본문은 기능별로 나눕니다. `요약`, `주요 구현`, `검증`만 있는
페이지는 읽기 쉽지만 판단하기에는 부족합니다. 최소한 기능별로 "무엇이 바뀌었는지", "사용자가
얻는 결과", "입력과 출력", "성공 상태와 실패 신호"를 함께 적습니다.

## 기능 추가/수정 판단 기준

문서를 고칠 때는 다음 기준을 사용합니다.

- 새 문서 추가: 기존 문서가 다루지 않는 독립적인 결정 주제가 있을 때.
- 기존 문서 수정: 역할은 맞지만 설명이 부족하거나 경로 중심이라 사용자가 판단하지 못할 때.
- 문서 통합: 같은 질문에 답하는 문서가 여러 개로 흩어져 있을 때.
- 문서 축소: 구현 세부사항이 설명보다 많아 읽기 흐름을 방해할 때.

## 검증 체크리스트

- 문서의 첫 30초 안에 “무엇을 하는지” 알 수 있는가?
- 사용자가 “왜 필요한지” 이해할 수 있는가?
- 실행 흐름이 단계로 설명되어 있는가?
- 입력과 출력, 성공 상태, 실패 신호가 함께 있는가?
- 결과물이 무엇인지 명시되어 있는가?
- 기능 추가/수정/보류를 판단할 기준이 있는가?
- 경로나 파일명이 본문 설명을 압도하지 않는가?
- Notion과 로컬 문서가 서로 다른 메시지를 말하지 않는가?
