# 기능 추가/수정 결정 가이드

## 한 문장 정의

이 문서는 새 기능을 추가할지, 기존 기능을 수정할지, 보류할지, 삭제할지를 결정하기 위한 **상위 canonical 기준**입니다. 워크플로 카탈로그, 에이전트 레지스트리, 문서화 스타일 가이드 등의 개별 "기능 추가/수정 판단 기준" 섹션은 이 문서의 원칙을 각 도메인에 구체화한 것이며, 충돌 시 이 문서가 우선합니다.

## 무엇을 하는가

기능 요청이나 개선 아이디어가 들어왔을 때 바로 구현하지 않고 먼저 결정 구조를 만듭니다. 이 가이드는 기능의 목적, 사용자 가치, 입력, 출력, 검증 방법, 유지 비용을 확인해 다음 행동을 고르게 합니다.

가능한 결정은 네 가지입니다.

- 추가: 기존 시스템에 없는 책임이 필요합니다.
- 수정: 책임은 이미 있지만 설명, 흐름, 결과, 검증이 부족합니다.
- 보류: 목표나 성공 기준이 아직 불명확합니다.
- 삭제 또는 축소: 가치보다 유지 비용이 크거나 중복 책임이 있습니다.

## 왜 필요한가

기능을 많이 추가한다고 시스템이 좋아지는 것은 아닙니다. 사용자에게 필요한 것은 더 많은 구조가 아니라 더 나은 판단 가능성입니다. 어떤 기능이 필요한지 결정하려면 “무엇을 해결하는가”와 “어떻게 성공을 증명하는가”가 먼저 있어야 합니다.

이 가이드는 기능 추가가 문서 복잡도만 늘리는 상황을 막고, 기존 기능 수정으로 해결할 수 있는 문제를 새 기능으로 오판하지 않게 합니다.

## 어떻게 동작하는가

기능 요청은 아래 순서로 검토합니다.

1. 사용자가 겪는 문제를 한 문장으로 적습니다.
2. 현재 시스템이 이미 그 문제를 해결하는지 확인합니다.
3. 해결한다면 설명, 접근성, 검증 기준이 부족한지 봅니다.
4. 해결하지 못한다면 새 책임이 필요한지 확인합니다.
5. 새 책임이라면 입력, 출력, 성공 기준, 실패 기준을 정합니다.
6. 가장 싼 검증 방법을 먼저 정합니다.
7. 추가, 수정, 보류, 삭제 중 하나로 결정합니다.

## 결과는 무엇인가

이 가이드를 적용한 기능 검토는 다음 결과를 남깁니다.

- 결정: 추가, 수정, 보류, 삭제 중 하나
- 이유: 왜 그 결정이 맞는지에 대한 짧은 근거
- 기대 결과: 사용자가 무엇을 더 잘 할 수 있게 되는지
- 검증 방법: 가장 작은 확인 방법
- 영향 범위: 어떤 문서, 규칙, 워크플로, 에이전트에 영향을 주는지
- 다음 행동: 바로 구현, 문서 수정, 사용자 질문, 검증 실행 중 하나

## 판단 기준

### 추가한다

다음 조건이 모두 맞으면 새 기능을 추가합니다.

- 기존 기능이 담당하지 않는 독립적인 사용자 목표가 있습니다.
- 입력과 출력이 설명 가능합니다.
- 성공 기준이 검증 가능합니다.
- 다른 기능을 불필요하게 복잡하게 만들지 않습니다.
- 반복 사용 가능성이 있습니다.

### 수정한다

다음 조건이면 기존 기능을 수정합니다.

- 현재 기능의 책임은 맞습니다.
- 사용자가 무엇을 하는 기능인지 이해하지 못합니다.
- 결과와 검증 기준이 문서에 없습니다.
- 경로, 파일명, 내부 구조 설명이 사용자 판단을 방해합니다.
- 실행 흐름은 있지만 왜 그런 흐름인지 설명이 부족합니다.

### 보류한다

다음 조건이면 보류합니다.

- 해결하려는 문제가 명확하지 않습니다.
- 성공 기준을 아직 말할 수 없습니다.
- 필요한 사용자나 상황이 특정되지 않았습니다.
- 검증 방법 없이 구현만 하게 됩니다.
- 기존 기능과의 경계가 불분명합니다.

### 삭제하거나 축소한다

다음 조건이면 삭제 또는 축소를 검토합니다.

- 같은 책임을 더 단순한 기능이 이미 수행합니다.
- 유지 비용이 실제 사용 가치보다 큽니다.
- 사용자가 읽어도 행동으로 이어지지 않습니다.
- 자동화가 인간 승인이나 검증 없이 핵심 문서를 바꿉니다.
- 문서가 시스템 이해보다 내부 구조 노출에 치우칩니다.

## 질문 템플릿

기능을 검토할 때 아래 질문에 답합니다.

- 사용자는 지금 무엇을 결정하지 못하고 있는가?
- 이 기능은 그 결정을 어떻게 쉽게 만드는가?
- 지금 있는 기능을 고치면 해결되는가?
- 새 기능이라면 누가 언제 사용하게 되는가?
- 성공한 결과를 무엇으로 확인할 수 있는가?
- 실패하면 어떤 손해가 생기는가?
- 가장 작게 검증하는 방법은 무엇인가?

## 예시

문제가 “Notion 문서를 읽어도 어떤 기능을 추가해야 할지 모르겠다”라면 새 자동화 기능을 먼저 만들 필요가 없습니다. 기존 문서화 규칙을 수정해 모든 문서가 무엇, 왜, 어떻게, 결과, 판단 기준을 설명하게 만드는 것이 먼저입니다. 그 뒤에도 반복적으로 누락된다면 문서 품질 검증 스크립트나 Notion 체크리스트를 새 기능으로 추가할 수 있습니다.
