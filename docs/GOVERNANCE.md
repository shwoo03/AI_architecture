# 에이전트 거버넌스

## 한 문장 정의

거버넌스는 에이전트가 무엇을 읽고, 무엇을 제안하고, 무엇을 직접 바꿀 수 있는지 정하는 승인과 감사 기준입니다.

## 무엇을 하는가

이 문서는 에이전트 행동의 경계를 정합니다. 에이전트가 독립적으로 조사할 수 있는지, 제안만 해야 하는지, 제한된 범위에서 쓸 수 있는지, 사람이 승인해야 하는지를 구분합니다. 또한 어떤 사건을 반드시 로그로 남겨야 하는지도 정합니다.

## 왜 필요한가

에이전트가 편리하다는 이유로 핵심 문서, 지식, 권한, 자동화를 임의로 바꾸면 시스템 신뢰도가 떨어집니다. 거버넌스는 자동화의 속도보다 추적 가능성과 승인 가능성을 우선하기 위해 필요합니다.

## 어떻게 동작하는가

권한은 네 단계로 나눕니다.

| 권한 | 의미 | 기본 산출물 |
| --- | --- | --- |
| 읽기 | 승인된 자료를 조사합니다. | 발견 사항과 요약 |
| 제안 | 변경안을 만들지만 적용하지 않습니다. | 제안 문서 |
| 쓰기 | 지정된 범위 안에서 수정합니다. | 제한된 파일 변경 |
| 승인 | 제안을 공식 반영합니다. | 기본적으로 사람만 수행 |

핵심 문서, 장기 지식, 권한 규칙, 자동화 적용 모드, 증거 삭제는 사람이 승인해야 합니다. 에이전트는 필요한 경우 제안을 만들 수 있지만, 승인 없이 핵심 기준을 바꾸지 않습니다.

권한 결정은 `config/policy.yaml`의 typed permission policy를 따른다. 기본 action은 `ask`, 시간 초과는 `deny`, 허용 가능한 결정은 `allow_once`, `allow_session`, `deny`다. 세션/프로젝트 범위 허용은 감사 가능한 이벤트로 남기며, wildcard shell 실행은 자동 허용하지 않는다.

## 결과는 무엇인가

거버넌스를 적용하면 다음 상태를 유지합니다.

- 에이전트의 쓰기 범위가 명확합니다.
- 자동화가 제안과 적용을 구분합니다.
- 핵심 문서는 사람이 소유합니다.
- 활동과 검증 결과가 감사 가능하게 남습니다.
- 권한 초과 상황에서 멈추고 사용자에게 확인합니다.

## 반드시 기록할 사건

- 에이전트 실행 시작과 종료
- 주요 도구 사용과 결과
- 작업 체크아웃 시작, 커밋, 롤백
- 검증 결과
- 권한 거부
- 에스컬레이션 또는 피벗 조건
- 제안 생성, 승인, 거절, 적용

## 기능 추가/수정 판단 기준

거버넌스 기능을 바꿀 때는 다음 기준을 사용합니다.

- 새 권한이 필요하면 먼저 어떤 위험을 줄이는지 설명합니다.
- 자동화가 직접 적용으로 바뀌려면 dry-run 검증과 사람 승인이 필요합니다.
- 로그가 늘어난다면 실제 의사결정에 쓰이는지 확인합니다.
- 핵심 문서 변경 권한은 기본적으로 사람에게 둡니다.
- 에이전트 권한 확장은 작업 범위와 검증 기준이 명확할 때만 허용합니다.

## 에스컬레이션 기준

다음 상황에서는 사용자 검토로 올립니다.

- 요청이 에이전트의 쓰기 범위를 넘습니다.
- 검증 결과가 성공 기준과 충돌합니다.
- 반복 실패가 피벗 기준에 도달했습니다.
- 반복 워크플로가 레지스트리에 등록되지 않았습니다.
- 문서나 지식의 출처가 불명확합니다.

## 사용자와 에이전트의 실행 분담

이 프로젝트에서 사용자는 셸 명령을 대신 실행하는 운영자가 아닙니다. 사용자는 목표, 제약, 승인 여부, 방향 수정을 채팅으로 제공합니다. 에이전트는 승인된 범위 안에서 필요한 스크립트 실행, 파일 수정, 검증, 기록 갱신을 직접 수행합니다.

사용자에게 명령 실행을 넘기는 것은 예외입니다. 로컬 권한, 네트워크, 계정, 비밀값 접근처럼 에이전트가 실제로 수행할 수 없는 경우에만 어떤 작업이 막혔는지와 사용자가 제공해야 할 최소 정보를 설명합니다. 일반적인 검증 명령, 장부 기록, 활동 로그, 인수인계 갱신은 에이전트 책임입니다.
