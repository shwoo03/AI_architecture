# Notion 문서화 규칙

## 이 문서는 선택 사항입니다

프로젝트가 Notion 워크스페이스에 문서를 미러링할 때만 이 규칙을 따릅니다. Notion을 쓰지 않는 프로젝트는 이 문서와 `AGENTS.md`, `CLAUDE.md`의 Notion 관련 규칙을 무시해도 됩니다. 로컬 Markdown만으로도 skeleton 운영은 완전합니다.

## 한 문장 정의

Notion 문서는 사용자가 시스템 구조를 외우게 만드는 곳이 아니라, 기능의 역할과 개선 결정을 이해하게 만드는 곳입니다.

## 독자 기준

Notion 문서의 독자는 해당 기능을 처음 보는 사람입니다. 에이전트는 독자가 코드 구조, 이전 대화,
오늘 작업 기록, 내부 파일명을 이미 안다고 가정하지 않습니다. 문서 하나만 열어도 다음 질문에 답할 수
있어야 합니다.

- 이 기능은 어떤 역할을 하는가?
- 왜 만들었는가?
- 언제 사용하는가?
- 어떤 입력을 받아 어떤 결과를 남기는가?
- 실제로는 어떤 순서로 동작하는가?
- 성공하면 무엇이 보이고, 실패하면 어떤 신호가 보이는가?
- 기능을 추가, 수정, 보류하려면 어떤 기준을 써야 하는가?

이 질문에 답하지 못하는 문서는 완료 문서가 아니라 보강 대상입니다.

## 무엇을 하는가

이 규칙은 공용 AI 아키텍처를 Notion에 문서화할 때 페이지를 어떻게 만들고, 어떤 내용을 앞에 두고, 어떤 정보를 보조로 숨길지 정합니다. 모든 Notion 문서는 데이터베이스 페이지로 관리하며, 본문은 기능 설명과 의사결정 기준을 중심으로 작성합니다.

## 왜 필요한가

경로, 파일명, 내부 폴더 구조를 앞세운 문서는 실제 사용자가 거의 읽지 않습니다. 사용자가 필요한 것은 “이 기능이 무엇을 해결하는지”, “왜 있는지”, “어떻게 동작하는지”, “결과가 무엇인지”, “무엇을 바꾸면 되는지”입니다.

Notion 문서가 이 질문에 답하지 못하면 기능 추가나 수정 결정을 지원하지 못합니다. 이 규칙은 Notion을 내부 파일 지도에서 운영 판단 문서로 바꾸기 위해 필요합니다.

## 사용자가 겪는 문제

Notion 문서가 날짜별 구현 기록이나 파일 경로 목록으로 작성되면 사용자는 실제로 무엇이 좋아졌는지
판단하기 어렵습니다. "어떤 파일을 고쳤다"는 정보만으로는 다음 기능을 추가해야 하는지, 기존 기능을
수정해야 하는지, 아니면 검증을 먼저 해야 하는지 결정할 수 없습니다.

또 다른 문제는 데이터베이스 보기입니다. 기본 view가 이름과 상태만 보여주면 문서가 기능별로
정리되어 있는지 알 수 없습니다. 사용자는 페이지를 하나씩 열어야만 Crawler, Replay, Verify,
Web UI 같은 기능 영역을 파악할 수 있고, 그 과정에서 최신 기준과 오래된 구현 기록이 섞여 보입니다.

## 어떻게 동작하는가

Notion에서는 데이터베이스의 각 페이지를 하나의 문서 단위로 사용합니다. 계층, 상태, 문서 유형, 담당 기능 같은 구조 정보만 데이터베이스 속성으로 관리하고, “무엇/왜/어떻게/결과/판단 기준” 같은 설명은 반드시 페이지 본문에 작성합니다.

문서 본문은 `docs/DOCUMENTATION_STYLE_GUIDE.md`의 필수 7-섹션 + 선택 3-섹션 구조를
따릅니다. Notion 고유 항목은 그 위에 덧붙입니다. 단, Notion에 올리는 기능 문서는
선택 확장을 사실상 필수로 취급합니다. 기능 기준으로 읽히지 않는 짧은 구현 기록, 변경 목록,
파일 목록은 완료 문서가 아니라 작업 로그 또는 초안입니다.

- 필수 7-섹션: 한 문장 정의, 무엇을 하는가, 왜 필요한가, 어떻게 동작하는가, 결과는 무엇인가,
  기능 추가/수정 판단 기준, 구현 연결 정보.
- 기능 페이지, 프로젝트 허브, 운영 가이드, 결정 페이지에서는 선택 확장(사용자가 겪는 문제,
  입력과 출력, 실패 신호)을 추가합니다.
- Notion 고유 추가 섹션: **하위 항목 읽는 순서** (하위 페이지가 있을 때만).

경로나 파일명은 첫 화면에 노출하지 않습니다. 필요하면 마지막의 구현 연결 정보에만 적습니다.

## 상세 본문 템플릿

핵심 기능 페이지는 다음 질문에 답할 수 있을 만큼 자세히 작성합니다. 짧은 답이 아니라, 사용자가 기능 추가나 수정을 결정할 수 있는 수준의 설명을 제공합니다.

- `한 문장 정의`: 이 기능을 모르는 사람이 한 번에 역할을 이해할 수 있게 씁니다.
- `사용자가 겪는 문제`: 이 기능이 없을 때 사용자가 어떤 판단을 못 하거나 어떤 실수를 하는지 씁니다.
- `무엇을 해결하는가`: 기능이 책임지는 범위와 책임지지 않는 범위를 함께 씁니다.
- `왜 필요한가`: 내부 구현 편의가 아니라 사용자 판단, 복원, 검증, 감사 관점에서 이유를 씁니다.
- `사용 시점`: 어떤 상황에서 이 기능이나 문서를 먼저 봐야 하는지 씁니다.
- `어떻게 동작하는가`: 실제 흐름을 4~7단계로 씁니다. 표는 역할 비교나 단계 요약에만 사용합니다.
- `입력과 출력`: 이 기능이 읽는 정보와 남기는 결과물을 씁니다.
- `성공 상태`: 사용자가 무엇을 확인하면 이 기능이 잘 동작한다고 볼 수 있는지 씁니다.
- `실패 신호`: 문서를 고치거나 기능을 수정해야 하는 증상을 씁니다.
- `기능 추가/수정/보류 판단 기준`: 새 기능 추가, 기존 기능 수정, 보류를 구분하는 기준을 씁니다.
- `하위 항목 읽는 순서`: 하위 항목이 있으면 어떤 순서로 읽어야 전체 구조가 이해되는지 씁니다.
- `구현 연결 정보`: 경로, 파일명, 스크립트, 런타임 로그 위치는 마지막에만 둡니다.

## 기능 중심 작성 계약

Notion 페이지는 파일이나 날짜가 아니라 기능을 기준으로 제목과 본문을 구성합니다. 사용자가
페이지를 열었을 때 "이 기능을 계속 키워야 하는가, 고쳐야 하는가, 보류해야 하는가"를 판단할 수
있어야 합니다.

모든 기능 페이지는 아래 골격을 기본값으로 사용합니다. 이름은 도메인에 맞게 조금 바꿀 수 있지만,
역할은 빠지면 안 됩니다.

1. `한 문장 정의`: 처음 보는 사람이 기능의 정체를 바로 알 수 있게 씁니다.
2. `사용자가 겪는 문제`: 이 기능이 없으면 어떤 판단이나 작업이 막히는지 씁니다.
3. `무엇을 하는가` 또는 `무엇을 해결하는가`: 책임 범위와 하지 않는 일을 함께 씁니다.
4. `왜 필요한가`: 내부 구현 편의가 아니라 사용자 이해, 복구, 검증, 감사 관점으로 씁니다.
5. `사용 시점`: 사용자가 언제 이 기능이나 문서를 먼저 봐야 하는지 씁니다.
6. `어떻게 동작하는가`: 실제 흐름을 번호 목록으로 씁니다.
7. `입력과 출력`: 읽는 정보와 남기는 결과물을 씁니다.
8. `성공 상태`: 사용자가 무엇을 보면 정상 동작이라고 판단할 수 있는지 씁니다.
9. `실패 신호`: 문서나 구현을 다시 봐야 하는 증상을 씁니다.
10. `기능 추가/수정/보류 판단 기준`: 다음 변경 결정을 위한 기준을 씁니다.
11. `구현 연결 정보`: 경로, 스크립트, 테스트, 로그 위치를 마지막에 둡니다.

- 제목은 기능 또는 결정 주제를 앞세웁니다. 날짜는 필요할 때 뒤에 붙입니다.
- 첫 화면에는 기능의 목적, 사용자가 겪는 문제, 현재 결정 상태가 보여야 합니다.
- "구현했다", "수정했다", "파일을 추가했다"로 끝내지 않고, 그 변경이 사용자 흐름과 산출물에
  어떤 차이를 만드는지 설명합니다.
- 기능 영역이 여러 개면 본문을 기능별 섹션으로 나누고, 각 섹션마다 역할, 입력, 출력, 성공 상태,
  실패 신호를 씁니다.
- 세부 경로, 테스트 명령, 산출물 파일명은 `구현 연결 정보`에 모읍니다. 본문 앞부분은 기능
  이해를 방해하지 않아야 합니다.
- 당일 구현 기록을 남길 때도 단순 changelog가 아니라 "기능별 변경 요약 -> 사용자가 얻는 결과
  -> 검증 -> 다음 판단" 순서로 씁니다.

다음 문서는 불완전합니다.

- 본문 대부분이 파일 경로, 테스트 명령, 점수 표인 문서.
- "요약", "주요 구현", "검증"만 있고 기능의 문제/입력/출력/실패 신호가 없는 문서.
- 기능 영역 속성은 채웠지만 본문에서 기능별로 나누어 설명하지 않은 문서.
- 코드 변경은 설명하지만 사용자가 다음에 무엇을 결정해야 하는지 말하지 않는 문서.

## 규칙처럼 동작하는 작성 절차

Notion 문서화는 자유 작문이 아니라 반복 가능한 운영 절차로 다룹니다.

1. 대상 DB를 fetch해 스키마와 view를 확인합니다.
2. 같은 기능을 다루는 기존 페이지를 검색합니다.
3. 새 페이지가 아니라 기존 페이지 업데이트가 맞는지 먼저 판단합니다.
4. 기능별 Markdown 초안을 위 골격으로 작성합니다.
5. 초안에서 경로와 테스트 명령이 앞부분을 차지하면 `구현 연결 정보`로 옮깁니다.
6. 가능한 경우 `python scripts/notion-doc-quality-check.py <draft.md>`를 실행합니다.
7. Notion에 작성 또는 교체한 뒤 fetch로 제목, 속성, 본문이 실제 반영됐는지 확인합니다.
8. 활동 로그에 `notion_sync` 또는 문서 업데이트 이벤트를 남깁니다.

이 절차를 건너뛰고 바로 페이지를 만들면 중복 문서, 얕은 구현 기록, 경로 중심 문서가 다시 생깁니다.

## 결과는 무엇인가

이 규칙을 적용하면 Notion 문서는 다음 상태가 됩니다.

- 사용자가 페이지를 열면 즉시 기능의 목적을 이해합니다.
- 기능 추가, 수정, 보류, 삭제 판단 기준을 볼 수 있습니다.
- 데이터베이스 뷰는 경로가 아니라 상태, 계층, 기능, 문서 유형 중심으로 보입니다.
- 세부 구현 위치는 보조 정보로만 남습니다.
- 로컬 문서와 Notion 문서가 같은 설명 구조를 공유합니다.

## 입력과 출력

입력은 사용자의 문서화 요청, 대상 Notion 데이터베이스, 기존 페이지 검색 결과, 로컬 문서와 검증
로그입니다. 페이지를 새로 만들거나 업데이트하기 전에는 같은 기능을 다루는 기존 페이지가 있는지
확인해야 합니다.

출력은 기능 기준으로 읽히는 Notion 데이터베이스 페이지입니다. 페이지 본문은 기능의 문제, 해결 범위,
동작 흐름, 입력과 출력, 성공 상태, 실패 신호, 기능 추가/수정/보류 판단 기준을 담습니다. 데이터베이스
보기는 `기능별 문서`를 기본으로 두고, 기능 영역과 문서 유형을 첫 화면에서 확인할 수 있어야 합니다.

## 성공 상태

Notion 문서화가 성공한 상태는 사용자가 페이지를 열고 다음 판단을 바로 할 수 있는 상태입니다.
기능이 무엇을 해결하는지, 어떤 입력을 읽고 어떤 결과를 남기는지, 어떤 신호가 보이면 수정해야
하는지, 다음에 어떤 문서를 읽어야 하는지가 본문에 드러나야 합니다.

기본 데이터베이스 view도 성공 상태에 포함됩니다. 사용자가 링크를 열었을 때 `기능별 문서` view에서
`이름`, `문서 유형`, `기능 영역`, `상태`, `최근 검토일` 같은 최소 속성을 볼 수 있어야 합니다. 코드
경로나 관계 속성은 관리용으로 남길 수 있지만 기본 읽기 흐름을 차지하면 안 됩니다.

## 사용 시점

이 규칙은 Notion에 새 문서를 만들거나 기존 문서를 업데이트할 때마다 사용합니다. 특히 다음 상황에서는
반드시 먼저 적용합니다.

- 사용자가 "노션 문서화", "오늘 작업 정리", "구현 내용 정리"를 요청했을 때.
- 기존 Notion 페이지가 날짜별 구현 기록이나 파일 목록 중심으로 되어 있을 때.
- 기능 영역이 여러 개인데 본문이 기능별로 나뉘어 있지 않을 때.
- 새로운 사람이 문서만 읽고 기능의 역할과 다음 행동을 알 수 없는 상태일 때.

## 데이터베이스 사용 규칙

- 문서는 데이터베이스 페이지로만 만듭니다.
- 페이지 안에 자식 페이지를 만들지 않습니다.
- 관련 문서는 관계 속성으로 연결합니다.
- 상태는 사용자가 문서 신뢰도를 판단할 수 있게 관리합니다.
- 경로 속성은 남길 수 있지만 기본 보기에서는 숨깁니다.
- 데이터베이스 속성은 제목, 계층, 문서 유형, 상태, 기능 요약, 관계, `Superseded By`, `Superseded At`처럼 정렬과 필터에 필요한 메타데이터만 둡니다.
- `무엇`, `왜`, `어떻게`, `동작`, `결과`, `수정 판단` 같은 설명형 속성을 만들지 않습니다. 이 내용은 페이지 본문 섹션으로 작성합니다.
- 새 문서를 만들 때 제목, 계층, 문서 유형, 상태, 기능 요약을 채우고 상세 설명은 본문에 작성합니다.

## 단일 기본 보기 규칙

- 사용자가 기본으로 읽는 Notion 데이터베이스 보기는 `기능별 문서` 하나로 둡니다.
- `기능별 문서` 보기는 모든 문서 유형을 포함하고, `이름` 기준 오름차순으로 정렬합니다.
- 기본 보기에서 표시하는 속성은 `이름`, `문서 유형`, `기능 영역`, `상태`, `최근 검토일`처럼 기능 이해에 필요한 최소 속성만 사용합니다. 영문 스키마를 쓰는 DB에서는 같은 의미의 `Name`, `Doc Type`, `Feature`, `Status`, `Reviewed At`을 사용합니다.
- `Directory`, `상위 항목`, `하위 항목` 같은 내부 구조 속성은 관계와 관리용으로 유지할 수 있지만 기본 보기에는 노출하지 않습니다.
- 추가 보기는 만들지 않는 것을 기본으로 합니다. 이미 존재하는 보조 보기는 간결한 이름으로 두되, 사용자에게 기준 보기는 `기능별 문서`라고 안내합니다.
- 도구가 view 삭제를 지원하지 않으면 view 삭제는 Notion UI에서 처리합니다.
- 보기 설정 도구가 native sub-item 펼침/접힘 옵션을 지원하지 않으면, 관계 데이터는 유지하고 Notion UI에서 Sub-items 표시를 수동 확인합니다.

## 본문 작성 규칙

- 한국어를 기본 언어로 사용합니다.
- 도메인 예시는 보조 예시로만 사용합니다.
- 모든 설명은 임의의 프로젝트에 적용 가능한 범용 설명이어야 합니다.
- 사용자가 다음 행동을 결정할 수 없는 문서는 불완전한 문서로 봅니다.
- 본문이 400자 이하인 핵심 기능 페이지는 기본적으로 불완전한 문서로 봅니다. 아주 작은 기능이라도 문제, 해결 범위, 입력/출력, 성공/실패 신호를 적으면 이보다 길어지는 것이 정상입니다.
- 표는 비교와 상태 요약에만 사용합니다.
- 경로 목록을 문서의 핵심 내용으로 쓰지 않습니다.
- 핵심 기능 페이지는 문제, 해결 범위, 입력/출력, 성공 상태, 실패 신호를 반드시 포함합니다.
- 하위 항목 페이지는 부모 기능 안에서의 역할, 사용 시점, 성공/실패 신호를 포함합니다.

## 중복 방지 프로토콜

같은 주제의 "최신판"과 "구판"이 동시에 공존하는 상태를 만들지 않습니다. 이미 다룬 주제를 다시 쓸 가치가 있으면 기존 페이지를 **업데이트**하고, 전면 재작성이 필요하면 **대체 처리**를 끝낸 후에만 새 페이지가 독립적으로 보이게 만듭니다. 다음 프로토콜을 새 Notion 페이지를 만들기 전에 **매번** 실행합니다.

### 1단계: 생성 전 검색 (필수)

새 페이지를 만들기 전에 데이터베이스를 반드시 검색합니다. `notion-search` / `notion-fetch` 같은 도구로 다음을 모두 시도합니다.

- 새 제목의 **핵심 명사**로 제목 검색 (예: "CTF 인스턴스 관리 도구").
- 같은 `Feature` 태그를 가진 페이지 나열.
- 같은 `Layer` 또는 `Doc Type` 안의 활성 페이지 훑어보기.

하나라도 결과가 있으면 **만들기 전에 사용자에게 묻습니다**. 임의로 새 페이지를 만들지 않습니다.

```text
기존 페이지 발견: "<제목>" (id=<page_id>, status=<status>).
다음 중 어느 행동을 할까요?
  (1) 기존 페이지 업데이트 (같은 주제의 개선/보강)
  (2) 새 페이지 생성 + 기존 페이지 대체 처리 (전면 재작성)
  (3) 새 페이지 생성 + 기존 페이지 유지 (주제가 실제로 다름)
```

사용자가 (3)을 선택하려면 "두 페이지가 다루는 범위가 어떻게 다른지"를 한 문장으로 설명하게 합니다. 설명이 안 나오면 (1) 또는 (2)로 돌아갑니다.

### 2단계: 업데이트 기본값 (1 선택 시)

기존 페이지 본문을 직접 교체합니다. 제목·속성·관계는 그대로 두고 본문만 새 설명으로 덮습니다. 업데이트 사실은 `runtime/activity-log.jsonl`에 `phase=notion_sync`, `action=page_updated`, `data={page_id, title}` 이벤트로 남깁니다. 새 페이지를 만들지 않습니다.

### 3단계: 대체 처리 (2 선택 시)

전면 재작성이 정당화될 때만 새 페이지를 만들고, **같은 작업 안에서** 구 페이지를 대체 상태로 마감합니다. 다음 네 가지를 모두 끝낸 뒤에야 작업이 완료입니다.

1. 새 페이지를 생성하고 id를 얻습니다.
2. 구 페이지의 `Superseded By` 관계 속성에 새 페이지를 연결합니다(없으면 아래 속성 규칙대로 추가).
3. 구 페이지의 `Superseded At`에 오늘 날짜(YYYY-MM-DD)를 기록합니다.
4. 구 페이지의 `Status`를 `superseded`로 바꿉니다. 제목 앞에 `[대체됨]` 접두사는 붙이지 않습니다(관계와 상태로 관리하며, 검색 결과에서 구 페이지가 숨어 오해를 일으키지 않게).

대체 처리 사실은 `action=page_superseded`, `data={old_page_id, new_page_id}` 이벤트로 활동 로그에 남깁니다.

### 데이터베이스 속성 확장 (대체 관계용)

다음 속성이 없으면 한 번 추가합니다.

- `Superseded By`: 같은 데이터베이스를 가리키는 **Relation** 속성. 최대 1개 연결.
- `Superseded At`: **Date** 속성.

기존 기본 보기(`기능별 문서`)에는 이 두 속성을 표시하지 않습니다. 별도 필터 보기가 필요할 때만 노출합니다.

### 활성 문서 필터

`기능별 문서` 기본 보기는 `Status != superseded`를 암묵 기본으로 둡니다. 구 페이지는 DB에 남지만 일반 열람에서 숨겨집니다. 활성 문서 하나만 눈에 들어오게 하는 것이 목표입니다.

### 실패 신호

다음 상태가 보이면 이 프로토콜이 깨진 것입니다.

- 같은 `Feature` 태그 안에 `Status=active` 페이지가 둘 이상이고 둘 다 같은 범위를 설명.
- 새 페이지가 만들어졌지만 구 페이지의 `Superseded By`가 비어 있음.
- 구 페이지 `Status`가 여전히 `active`인데 본문에 "최종 문서로 대체됨" 같은 문장이 적혀 있음.

발견 시 이 프로토콜의 3단계를 소급 적용합니다.

## 기능 추가/수정 판단 기준

Notion 문서 자체를 바꿀 때는 다음 기준을 사용합니다.

- 새 페이지 추가: 기존 페이지가 다루지 않는 독립적인 판단 주제가 있을 때.
- 기존 페이지 수정: 같은 주제이지만 설명이 부족하거나 경로 중심일 때.
- 관계 추가: 두 문서가 같은 결정 흐름에 연결될 때.
- 보기 수정: 사용자가 먼저 봐야 할 속성이 경로보다 뒤에 있거나 숨겨져 있을 때.
- 보류: 문서 주제가 명확하지 않거나 기능 결과를 설명할 수 없을 때.

## 검증 체크리스트

- 첫 화면에서 “무엇을 하는 문서인지” 보이는가?
- 제목과 첫 섹션이 날짜나 파일명이 아니라 기능 또는 결정 주제를 앞세우는가?
- “왜 필요한지”가 내부 구현이 아니라 사용자 관점으로 설명되어 있는가?
- “어떻게 동작하는지”가 단계로 설명되어 있는가?
- 입력과 출력이 명확한가?
- 성공 상태와 실패 신호가 모두 있는가?
- 결과물이 명확한가?
- 기능 추가/수정 판단 기준이 있는가?
- 경로와 파일명이 본문 설명을 압도하지 않는가?
- 기능 영역이 여러 개라면 본문도 기능별로 나뉘어 있는가?
- 관련 문서가 데이터베이스 관계로 연결되어 있는가?
- 초안 파일이 있으면 `python scripts/notion-doc-quality-check.py <draft.md>`를 통과했는가?
- 새 페이지를 만들기 전에 DB를 검색해 유사 페이지가 없음을 확인했는가? (중복 방지 프로토콜 1단계)
- 같은 주제의 기존 페이지를 대체한다면 `Superseded By`·`Superseded At`·`Status=superseded` 세 필드가 모두 설정됐는가?
