# 운영 루프

## 한 문장 정의

운영 루프는 프로젝트를 목표 확인, 계획, 실행, 검증, 결정의 반복 가능한 흐름으로 진행하게 만드는 기본 절차입니다.

## 무엇을 하는가

이 문서는 에이전트가 어떤 순서로 프로젝트를 진행해야 하는지 정합니다. 각 단계에는 다음 단계로 넘어가기 위한 조건이 있습니다. 조건을 확인할 수 없으면 추측해서 진행하지 않고 사용자에게 확인합니다.

## 왜 필요한가

AI 프로젝트에서 가장 흔한 실패는 검증 없이 구현이 커지는 것입니다. 목표가 불명확한 상태에서 작업을 시작하거나, 성공 기준 없이 기능을 추가하거나, 실패한 결과를 기록하지 않으면 다음 세션은 같은 문제를 반복합니다.

운영 루프는 작업을 작게 만들고, 검증을 먼저 생각하게 하며, 각 결정의 증거를 남기기 위해 필요합니다.

## 어떻게 동작하는가

### 1. 프로필 확인

프로젝트 목표, 사용자, 성공 기준, 실패 기준, 제약 조건을 확인합니다. 프로필이 없으면 템플릿을 기준으로 사용자에게 질문해 만듭니다. 모르는 사실은 추측하지 않고 보류 표시를 남깁니다.

다음 단계 조건: 프로젝트 목표, 성공 기준, 실패 기준이 비어 있지 않아야 합니다.

### 2. 명세 작성

구현의 정답 조건이 필요할 때만 명세를 만듭니다. 명세에는 문제 정의, 인터페이스, 수용 테스트, 반례, 비목표를 적습니다.

다음 단계 조건: 수용 테스트가 숫자, 정확한 출력, 통과 횟수처럼 측정 가능한 결과를 가져야 합니다.

### 3. 계획

프로필과 명세를 바탕으로 가장 작은 실행 단위를 정합니다. 필요한 워크플로와 에이전트 역할을 고르고, 권한과 쓰기 범위를 확인합니다. 기본 에이전트 네 개는 canonical `agents/`에 정의되어 있습니다: `strategy-planner`(계획·트레이드오프), `codebase-explorer`(읽기 전용 조사), `implementation-worker`(배정 범위 구현), `independent-validator`(독립 검증). Codex/Claude 산출물은 `scripts/convert.py`로 생성합니다.

사용자가 `goal 프롬프트`를 요청하면 계획 산출물은 Codex CLI `/goal`에 넣을 짧은 프롬프트입니다. 이 프롬프트는 4000자 이하이어야 하며, 넘칠 경우 상세 계획을 문서로 분리하고 `/goal`은 해당 문서를 가리키는 진입점으로 작성합니다.

비판적 토론이 필요한 계획은 Claude 자동 토론이 아니라 subagent debate로 진행합니다. 이 방식은 사용자가 명시적으로 요청했거나, 범위가 크거나, 되돌리기 어렵거나, 근거 충돌이 있는 작업에 사용합니다. 일반적인 작은 수정은 평소처럼 Codex가 짧은 계획과 검증으로 진행합니다. Codex는 user-facing orchestrator로 남고, 서브 에이전트는 독립적인 비판·조사·검증 관점만 제공합니다. 토론 기록이 필요한 경우 `runtime/dialogues/*.jsonl`에 claim, critique, concession, convergence를 남기며, convergence는 Codex와 최소 1개 `subagent-*` readiness 및 열린 block critique 없음이 조건입니다.

다음 단계 조건: 가장 싼 검증 방법이 먼저 문서화되어야 합니다.

### 4. 실행

가장 작은 유용한 결과물을 만듭니다. 작업 범위를 넘기지 않고, 필요한 경우 에이전트 역할을 나누며, 반복 실행은 목표 계보와 실행 로그를 남깁니다.

기존 프로젝트의 구버전 뼈대를 최신화할 때는 release manifest 기준으로 진행합니다. `agent-flow adopt`로 read-only intake를 먼저 보고, `upgrade-from-skeleton --brief --profile stable`로 release id/source commit/channel과 변경 후보를 확인한 뒤, 승인된 `--apply --safe-only`만 자동 적용합니다. manual/risky 항목은 path별 승인 전까지 preserve 후보로 둡니다.

다음 단계 조건: 테스트할 수 있는 결과물이 있어야 하고, 주요 변경이 활동 로그에 기록되어야 합니다.

### 5. 검증

가장 싼 검증부터 실행합니다. 결과는 성공, 부분 진전, 시간 초과, 인프라 오류, 정책 차단처럼 분류합니다.

다음 단계 조건: 모든 검증 시도에 결과 분류가 기록되어야 합니다.

### 6. 결정

검증 결과를 바탕으로 계속할지, 수정할지, 재설계할지, 중단하고 회수할지 결정합니다.

다음 반복 조건: 결정이 로그나 피벗 기록으로 남아야 합니다.

## 결과는 무엇인가

운영 루프를 따르면 각 작업은 다음 결과를 남깁니다.

- 현재 목표와 성공 기준
- 실행할 최소 작업 단위
- 검증 방법
- 실행 결과
- 판단 근거
- 다음 행동

## 기능 추가/수정 판단 기준

운영 루프 안에서 기능 요청은 다음 기준으로 처리합니다.

- 목표와 성공 기준이 있으면 계획 단계로 보냅니다.
- 검증 방법이 없으면 실행하지 않습니다.
- 기존 기능의 설명이 부족한 문제라면 문서 수정으로 처리합니다.
- 반복되는 작업이면 워크플로 또는 에이전트 후보로 등록합니다.
- 검증 결과가 나쁘면 기능을 키우기 전에 실패 원인을 기록합니다.

## 루프 불변 조건

- 로그는 append-only입니다.
- 반복 에이전트 실행은 목표 계보를 포함합니다.
- 검증 전 주장은 장기 지식으로 확정하지 않습니다.
- 자동 생성 변경은 먼저 제안으로 둡니다.
- 세션 종료 전 인수인계를 갱신합니다.
- 단계 조건은 권장 사항이 아니라 진행 조건입니다.

## 자동 실행 주체

이 뼈대의 기본 실행자는 사용자나 운영자가 아니라 에이전트입니다. 문서에 적힌 `python3 scripts/...` 명령은 사용자가 직접 실행하라는 안내가 아니라, 에이전트가 작업 중 스스로 실행해야 하는 검증 계약입니다.

사용자는 목표, 제약, 승인, 거절, 방향 수정을 채팅으로 제공합니다. 에이전트는 그 의사를 바탕으로 필요한 탐색, 파일 수정, 검증 명령 실행, 활동 로그 기록, 세션 인수인계 갱신을 자율적으로 수행합니다. 에이전트가 실행할 수 없는 명령이나 권한 밖 작업만 사용자에게 명확히 보고합니다.

작업 완료 조건은 "사용자에게 실행하라고 안내했다"가 아니라 "에이전트가 가능한 검증을 실제로 실행하고 결과를 기록했다"입니다.
