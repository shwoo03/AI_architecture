# 피벗과 에스컬레이션 기준

## 한 문장 정의

피벗과 에스컬레이션 기준은 효과가 낮은 시도를 계속 붙잡지 않고, 수정, 교체, 재설계, 중단을 판단하게 만드는 기준입니다.

## 무엇을 하는가

이 문서는 작업이나 프로젝트가 원하는 결과를 내지 못할 때 언제 멈추고 방향을 바꿀지 정합니다. 단일 단계 문제인지, 하위 시스템 문제인지, 프로젝트 전체 문제인지 구분하고, 실패한 작업에서도 재사용 가능한 교훈을 남깁니다.

## 왜 필요한가

AI 프로젝트는 “조금만 더 하면 될 것 같다”는 이유로 실패한 접근을 오래 유지하기 쉽습니다. 명확한 피벗 기준이 없으면 시간과 비용이 늘어나고, 실패 이유가 기록되지 않아 같은 시도를 반복합니다.

## 어떻게 동작하는가

각 의미 있는 단계에는 효과 점수를 매깁니다. 점수는 0부터 10까지이며, 시간, 재시도 횟수, 오류율, 사용자 관점 품질, 외부 검증 성공 여부, 부분 진전 정도를 함께 봅니다.

기본 해석은 다음과 같습니다.

| 점수 | 의미 | 행동 |
| --- | --- | --- |
| 0-2 | 효과 없음 | 교체 또는 중단 후보 |
| 3-4 | 약함 | 대안 설계 필요 |
| 5-7 | 사용 가능 | 회귀 여부 관찰 |
| 8-10 | 강함 | 반복 성공 후 고정 후보 |

프로젝트별 임계값이 필요하면 운영 계획에서 따로 정합니다.

## 단계별 피벗 기준

| 신호 | 조건 | 행동 |
| --- | --- | --- |
| 같은 단계가 반복 실패 | 낮은 효과 점수가 연속됩니다. | 해당 단계를 교체합니다. |
| 변경 후 성능 회귀 | 이전보다 효과가 뚜렷하게 떨어집니다. | 회귀 원인을 분리하거나 되돌립니다. |
| 같은 실패 유형 반복 | 같은 원인이 반복됩니다. | 금지 패턴이나 경고를 제안합니다. |
| 검증이 선행 조건에 막힘 | 전제 조건이 실패합니다. | 검증을 계속하지 말고 전제를 먼저 고칩니다. |
| 예산 초과 | 단계가 허용 자원보다 큽니다. | 단순화, 분할, 제거를 선택합니다. |

## 프로젝트 수준 피벗 기준

| 신호 | 조건 | 행동 |
| --- | --- | --- |
| 반복해도 진짜 성공이 없음 | 여러 반복 뒤에도 핵심 성공이 없습니다. | 아키텍처를 재설계합니다. |
| 전체 검증이 계속 시간 초과 | 부분 진전 없이 반복됩니다. | 범위를 줄이거나 보관합니다. |
| 성공 기준이 측정 불가 | 무엇이 성공인지 말할 수 없습니다. | 프로젝트 프로필을 다시 정의합니다. |
| 비용 대비 가치가 낮음 | 시간이나 비용이 결과보다 큽니다. | 중단하고 회수 절차를 실행합니다. |
| 같은 에이전트 역할이 반복적으로 범위를 넘는다 | 작업 범위나 역할 정의가 잘못됐습니다. | 에이전트를 멈추고 역할·범위를 재정의한 뒤 사용자 승인 후 재개합니다. |

## 에스컬레이션 단계

| 단계 | 신호 | 결정 |
| --- | --- | --- |
| 1단계 | 한 단계의 효과가 정체됩니다. | 해당 단계를 교체합니다. |
| 2단계 | 교체도 반복 실패합니다. | 하위 시스템을 재설계합니다. |
| 3단계 | 반복 가치가 낮습니다. | 프로젝트 전체 계획을 다시 세웁니다. |
| 4단계 | 핵심 성공이 계속 없습니다. | 회수하고 보관합니다. |

## 결과는 무엇인가

이 기준을 적용하면 다음을 얻습니다.

- 계속할지 멈출지 감으로 결정하지 않습니다.
- 실패한 접근의 이유를 기록합니다.
- 작은 단계 교체와 전체 재설계를 구분합니다.
- 중단 전 재사용 가능한 교훈을 남깁니다.
- 예산과 검증 결과가 실제 의사결정에 쓰입니다.

## 기능 추가/수정 판단 기준

피벗 기준을 바꿀 때는 다음을 확인합니다.

- 새 기준이 실제 중단 또는 방향 전환 결정을 돕는가?
- 너무 엄격해 유효한 실험을 막지 않는가?
- 너무 느슨해 실패한 접근을 계속 허용하지 않는가?
- 검증 결과와 연결되는가?
- 프로젝트별 임계값은 공통 기준이 아니라 프로젝트 운영 계획에 둘 수 있는가?

## 회수 절차

프로젝트나 접근을 중단하기 전에 회수 기록을 남깁니다.

- 재사용할 만한 작동 패턴
- 실패한 가정
- 검증 증거
- 보존할 파일이나 프롬프트
- 다음에는 다르게 시도할 점

실패 증거는 삭제하지 않습니다. 보관해 두어야 다음 프로젝트가 같은 실패를 반복하지 않습니다.
