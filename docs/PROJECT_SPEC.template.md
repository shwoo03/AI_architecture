# 프로젝트 명세 템플릿

에이전트는 구현의 정답 조건이 필요할 때만 이 파일을 `docs/PROJECT_SPEC.md`로 복사합니다.

## 한 문장 정의

프로젝트 명세는 “무엇이 올바른 구현인가”를 판단하기 위한 수용 기준, 인터페이스, 반례, 검증 조건입니다.

## 무엇을 하는가

프로젝트 프로필이 프로젝트의 목적을 말한다면, 프로젝트 명세는 구현이 맞는지 판단하는 계약을 제공합니다. 명세는 계획, 구현, 검증 단계에서 에이전트가 자의적으로 성공을 해석하지 않게 합니다.

## 왜 필요한가

구현 작업은 “대충 동작한다”와 “정확히 성공했다”를 구분해야 합니다. 명세가 없으면 에이전트는 결과를 낙관적으로 해석하거나, 사용자가 기대하지 않은 동작을 성공으로 판단할 수 있습니다.

## 수용 테스트

좋은 `Measurable Outcome` 예시:

- `p95 latency < 500ms across 100 requests`

- `works correctly`

## 결과는 무엇인가

명세가 있으면 에이전트와 사용자는 다음을 알 수 있습니다.

- 무엇을 해결해야 하는지
- 어떤 입력과 출력이 계약인지
- 어떤 경우를 실패로 봐야 하는지
- 어떤 테스트를 통과해야 하는지
- 구현 결정이 어떤 가설을 검증하는지

## 기능 추가/수정 판단 기준

구현 결과의 성공 여부가 대화만으로 모호하면 명세를 만듭니다. 단순 조사, 문서 정리, 초기 탐색처럼 정답 조건이 아직 필요 없는 작업에는 명세를 만들지 않습니다. 수용 테스트를 측정 가능하게 쓸 수 없다면 먼저 목표와 성공 기준을 다시 질문합니다.
