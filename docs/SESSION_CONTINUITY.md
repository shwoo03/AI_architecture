# 세션 연속성 규칙

## 한 문장 정의

세션 연속성 규칙은 긴 작업이 끝나거나 컨텍스트가 줄어들 때 다음 세션이 같은 상태에서 이어받게 만드는 인수인계 기준입니다.

## 무엇을 하는가

이 규칙은 현재 작업 상태와 전체 실행 이력을 분리합니다. 현재 상태는 한 파일에 덮어쓰고, 실행 이력은 append-only 로그에 남깁니다. 다음 세션은 먼저 현재 상태를 읽고, 필요하면 최근 로그를 확인해 이어갑니다.

## 왜 필요한가

AI 작업은 세션이 끊기면 맥락을 잃기 쉽습니다. 대화 기록만 믿으면 마지막 결정, 열린 질문, 다음 행동, 바뀐 파일을 놓칠 수 있습니다. 세션 인수인계는 다음 에이전트가 추측하지 않고 바로 이어받게 하기 위해 필요합니다.

## 어떻게 동작하는가

두 종류의 기록을 함께 사용합니다.

| 기록 | 역할 | 관리 방식 |
| --- | --- | --- |
| 세션 인수인계 | 지금 어디까지 왔는지 보여주는 현재 상태 | 저장할 때마다 전체 덮어쓰기 |
| 활동 로그 | 어떻게 여기까지 왔는지 보여주는 감사 이력 | append-only |

인수인계는 빠른 재개를 위한 스냅샷입니다. 활동 로그는 전체 추적을 위한 증거입니다.

## 언제 저장하는가

다음 상황에서는 세션 인수인계를 덮어씁니다.

- 사용자가 세션 저장이나 종료를 요청합니다.
- 컨텍스트 한계나 압축이 가까워집니다.
- 30분 단위 작업이 끝났습니다.
- 큰 작업이나 주요 결정이 완료됐습니다.
- 다음 세션이 바로 이어받아야 할 상태가 생겼습니다.

트리거를 확인하고 갱신을 수행하는 주체는 **지금 실행 중인 에이전트**입니다. 별도 타이머나 훅이 강제하지 않으므로, 에이전트는 위 조건이 발생했는지 주기적으로 자가 점검하고, 해당되면 사용자에게 짧게 알린 뒤 인수인계를 저장합니다. 사용자가 명시적으로 요청하지 않더라도, 에이전트가 큰 작업을 완료했다면 스스로 저장합니다.

인수인계를 저장할 때는 활동 로그에도 짧은 이벤트를 남깁니다.

```json
{"ts":"<UTC>","phase":"session","action":"handoff_saved","project":"<name>","data":{"next_step":"<one line>"}}
```

## 무엇을 적는가

인수인계에는 다음을 적습니다.

- 현재 작업: 지금 무엇을 하고 있었는지 한두 문장으로 적습니다.
- 마지막 완료: 이번 세션에서 끝난 일을 짧게 적습니다.
- 다음 단계: 다음 세션이 바로 실행할 수 있게 번호로 적습니다.
- 열린 질문과 차단 요소: 추측으로 진행하면 안 되는 내용을 적습니다.
- 이번 세션에서 바뀐 파일: 다음 세션이 확인할 대상을 줄입니다.
- 핵심 결정: 왜 그런 방향을 택했는지 적습니다.

오래된 내용은 남겨두지 않습니다. 인수인계는 누적 문서가 아니라 현재 상태 문서입니다.

## 민감 데이터 규칙

인수인계는 프로젝트 저장소 안의 일반 파일입니다. git에 커밋되거나 다른 사람과 공유될 수 있으므로 다음을 **그대로 적지 않습니다**:

- CTF flag, 대회 제출용 key, 정답 값
- API key, 비밀번호, 인증 토큰, 서명 키
- 개인 식별 정보(PII), 연락처, 주소
- 제3자 시스템의 자격 증명

민감한 값이 세션 결과에 포함되면 다음 중 하나로 대체합니다.

- `[REDACTED]` 또는 `[REDACTED: flag for challenge 5]` 같은 라벨
- 해시나 앞 4자 prefix만 기록 (예: `flag_hash=abc123…`)
- 위치 참조만 남기기 (예: `flag는 ~/ctf/flags/pwn-03.txt에 저장`)

다음 세션이 실제 값을 다시 필요로 하면 사용자에게 질문하거나 적절한 저장 위치(환경 변수, 비밀 관리 도구, 별도 로컬 파일)에서 조회합니다.

## 어떻게 재개하는가

새 세션은 다음 순서로 읽습니다.

1. 세션 인수인계
2. 활동 로그 최근 항목
3. 프로젝트 프로필
4. 필요하면 프로젝트 명세

읽은 뒤에는 이전 세션이 무엇을 하던 중이었는지 한두 문장으로 요약하고, 다음 단계 1번을 제안합니다. 파일을 수정하기 전에는 사용자 의도와 현재 상태가 맞는지 확인합니다.

인수인계 파일도 `Next step` 필드도 비어 있고 활동 로그도 bootstrap 이벤트만 있는 **신규 프로젝트 cold start**라면, 에이전트는 `docs/PROJECT_PROFILE.md`(있는 경우 그 빈 필드)와 `docs/PROJECT_PROFILE.template.md`의 "대화 질문 순서"를 참고해 사용자에게 첫 질문을 던지고 **답을 받을 때까지 기다립니다.** PROFILE.md 자체가 없으면(수동 복사 케이스) 템플릿을 기반으로 `docs/PROJECT_PROFILE.md`를 먼저 만듭니다. 추측으로 필드를 채우지 않고, 사용자 응답이 있을 때만 다음 행동을 결정합니다.

`resume-readiness.py --strict`는 handoff, activity log, completion evidence뿐 아니라 ownership lock 상태도 확인합니다. 새 파일 추가나 삭제 때문에 `runtime/ownership-classification.lock.json`에 addition/removal drift가 생기면 다음 세션이 놓치지 않도록 경고합니다. 세션 종료 전에 `python3 scripts/ownership-lock.py write`로 lock을 갱신하는 것이 기본입니다. 이번 세션에서 의도적으로 미루는 경우에는 인수인계 본문에 `ownership lock drift: deferred`를 명시하고, 다음 단계에 갱신 이유와 수행 명령을 적습니다.

## Machine-local path 정책 보류

durable state에 머신 전용 절대경로를 어떻게 남길지는 아직 정책 결정이 필요합니다. 구현 전 선택지는 다음 셋입니다.

1. `state/*`와 `runtime/state/*` 모두 `$HOME/...` 형태로 치환한다.
2. `state/*`는 portable하게 강제하고, `runtime/*`는 디버깅 증거로 exact path를 허용한다.
3. `portability-scan` allowlist로 예외를 관리한다.

정책이 확정될 때까지 scanner와 closeout 문서는 현재 동작을 유지하고, 새 규칙을 임의로 적용하지 않습니다.

## 결과는 무엇인가

세션 연속성을 지키면 다음 효과가 생깁니다.

- 다음 세션이 최근 대화만 보고 추측하지 않습니다.
- 열린 질문이 사라지지 않습니다.
- 바뀐 파일과 핵심 결정이 남습니다.
- 긴 작업을 안전하게 나눌 수 있습니다.
- 컨텍스트 압축 이후에도 작업 목표가 유지됩니다.

## 기능 추가/수정 판단 기준

세션 연속성 기능을 바꿀 때는 다음 기준을 사용합니다.

- 새 필드는 다음 세션의 실제 행동을 돕는가?
- 중복 기록을 늘리지 않는가?
- 사람이 읽고 바로 이어갈 수 있는가?
- 자동화가 생겨도 수동 인수인계와 충돌하지 않는가?
- 오래된 상태를 보존하려면 인수인계가 아니라 활동 로그에 남기는가?

## 실패 상황과 복구

| 상황 | 복구 방식 |
| --- | --- |
| 인수인계를 저장하지 못하고 종료됨 | 최근 활동 로그를 바탕으로 복원합니다. |
| 인수인계와 활동 로그가 충돌함 | 활동 로그를 우선하고 사용자에게 차이를 알립니다. |
| 인수인계가 삭제된 파일을 가리킴 | 조용히 재생성하지 말고 사용자에게 확인합니다. |
| 여러 세션이 병렬로 실행됨 | 각 세션의 기록을 비교하고 사용자가 병합합니다. |
| 활동 로그 마지막 줄이 개행 없이 잘림 (프로세스 중단, 디스크 풀 등) | `scripts/verify-skeleton.py`가 `activity_log_parse` 에러로 감지합니다. 다음 append는 OS 락이 해제되면 성공하지만 **깨진 마지막 줄에 덧붙어 한 줄에 두 JSON이 붙습니다**. 수동으로 해당 줄을 열고 잘린 조각을 지운 뒤 개행을 보장합니다. `verify-skeleton`이 다시 clean이 될 때까지 반복합니다. |
| `docs/PROJECT_PROFILE.md`가 존재하지만 0 바이트 | cold-start 규칙에 따라 에이전트가 템플릿을 기반으로 다시 시드하거나 사용자에게 첫 질문을 던집니다. `scripts/verify-skeleton.py`가 경고로 표시합니다. |

## 구현 연결 정보

- 세션 인수인계 파일: `runtime/state/session-handoff.md`
- 활동 로그: `runtime/activity-log.jsonl`
- 활동 로그 기록 스크립트: `scripts/hooks/post-tool-use-log.py` (자동 호출되지 않음; 에이전트가 수동으로 실행하거나 한 줄을 직접 append)
- 활동 로그 스키마: `docs/RUNTIME_EVENT_SCHEMA.md`
- 운영 루프: `docs/OPERATING_LOOP.md`
