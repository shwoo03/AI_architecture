# 스킬 로딩 모델

## 한 문장 정의

스킬 로딩 모델은 반복 작업 절차를 프로젝트 안에 두고, 필요할 때만 읽어 에이전트의 기본 맥락을 작게 유지하는 방식입니다.

## 무엇을 하는가

이 모델은 프로젝트별 스킬을 프로젝트와 함께 관리합니다. 개인 전역 스킬이나 외부 플러그인은 사용할 수 있지만, 이 공용 시스템이 자동으로 설치하거나 승격하지 않습니다. 기본 원칙은 단순합니다. 현재 프로젝트에서 반복적으로 쓰는 절차만 프로젝트 스킬로 둡니다.

## 왜 필요한가

항상 모든 스킬을 읽으면 에이전트의 맥락이 커지고, 관련 없는 절차가 판단을 흐립니다. 반대로 반복 작업을 매번 대화로 설명하면 일관성이 떨어집니다. 스킬 로딩 모델은 필요한 절차만 점진적으로 읽어 두 문제를 줄입니다.

## 어떻게 동작하는가

스킬은 세 단계로 읽힙니다.

1. 이름과 설명으로 사용할지 판단합니다.
2. 필요할 때 스킬 본문을 읽습니다.
3. 본문이 가리키는 세부 자료는 실제로 필요할 때만 읽습니다.

새 스킬은 반복 작업이 확인될 때만 추가합니다. 한 번만 쓰는 절차나 아직 검증되지 않은 아이디어는 스킬로 만들지 않고 제안이나 작업 로그에 남깁니다.

## 결과는 무엇인가

이 모델을 사용하면 다음 상태를 유지할 수 있습니다.

- 에이전트가 항상 읽는 문맥이 작습니다.
- 반복 작업은 같은 절차로 실행됩니다.
- 프로젝트별 스킬은 프로젝트와 함께 복사됩니다.
- 개인 전역 스킬은 사용자 소유로 남습니다.
- 플러그인은 팀 배포나 버전 관리가 필요할 때만 사용합니다.

## 기능 추가/수정 판단 기준

스킬을 만들거나 수정할 때는 다음 기준을 사용합니다.

- 새 스킬 추가: 같은 작업이 반복되고, 트리거 조건과 절차가 명확할 때.
- 기존 스킬 수정: 절차는 맞지만 누락된 검증, 체크리스트, 입력 조건이 있을 때.
- 스킬 보류: 한 번만 쓰는 작업이거나 성공 기준이 불명확할 때.
- 스킬 삭제 또는 축소: 실제로 사용되지 않거나 공통 규칙과 중복될 때.

## 운영 규칙

- 프로젝트별 스킬은 프로젝트 내부에 둡니다.
- 개인 전역 스킬은 사용자가 관리합니다.
- 공용 시스템은 전역 스킬 생명주기를 자동 관리하지 않습니다.
- 스킬 이름 충돌은 의도적인 프로젝트 오버라이드가 아니라면 피합니다.
- 스킬은 방법을 설명하고, 규칙은 반드시 지켜야 할 조건을 설명합니다.
