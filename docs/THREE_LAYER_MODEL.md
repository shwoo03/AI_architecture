# 세 층 AI 프로젝트 모델

## 한 문장 정의

세 층 모델은 공통 규칙, 재사용 가능한 스킬, 프로젝트별 사실을 분리해 AI 프로젝트가 커져도 구조가 섞이지 않게 하는 기준입니다.

## 무엇을 하는가

이 모델은 모든 내용을 한 문서나 한 프롬프트에 넣지 않습니다. 대신 “항상 참이어야 하는 규칙”, “필요할 때만 실행하는 절차”, “현재 프로젝트에만 해당하는 사실”을 분리합니다.

세 층은 다음과 같습니다.

- 범용 기반: 모든 프로젝트에 적용되는 원칙과 안전 경계입니다.
- 도메인 스킬: 특정 작업을 할 때만 읽는 절차와 체크리스트입니다.
- 프로젝트 오버레이: 현재 프로젝트의 목표와 제약 조건입니다.

## 왜 필요한가

프로젝트별 사실이 공통 규칙에 들어가면 다음 프로젝트에 잘못 복사됩니다. 반대로 공통 방법론이 프로젝트 프로필에 들어가면 프로필이 두꺼워져 에이전트가 중요한 목표를 놓칩니다. 세 층 모델은 이 둘을 분리해 정보의 위치를 결정하게 합니다.

## 어떻게 동작하는가

### 범용 기반

모든 프로젝트에서 변하지 않는 기준입니다. 에이전트가 어떻게 목표를 읽고, 권한을 확인하고, 검증하고, 기록할지 정합니다. 이 층은 자주 바꾸지 않습니다.

### 도메인 스킬

반복 업무가 확인될 때만 사용합니다. 예를 들어 테스트 주도 구현, 보안 리뷰, 검색 우선 조사, 검증 루프처럼 작업 절차가 있는 경우 스킬로 둡니다. 스킬은 항상 로드하지 않고, 조건이 맞을 때만 읽습니다.

### 프로젝트 오버레이

현재 프로젝트에만 해당하는 정보입니다. 목표, 사용자, 성공 기준, 실패 기준, 제약 조건, 사용하는 스택, 활성 워크플로를 담습니다. 오버레이는 얇게 유지합니다.

## 결과는 무엇인가

세 층 모델을 따르면 다음 효과가 생깁니다.

- 새 프로젝트에 불필요한 도메인 정보가 복사되지 않습니다.
- 에이전트가 항상 읽어야 할 문서가 작아집니다.
- 반복 업무는 스킬로 재사용됩니다.
- 프로젝트 목표는 얇고 명확하게 유지됩니다.
- 기능을 어디에 추가해야 하는지 판단하기 쉬워집니다.

## 기능 추가/수정 판단 기준

새 내용을 추가할 위치는 다음 기준으로 정합니다.

- 모든 프로젝트에 적용되는 불변 조건이면 범용 기반에 둡니다.
- 특정 작업을 수행하는 방법이면 도메인 스킬에 둡니다.
- 현재 프로젝트의 사실이나 예외라면 프로젝트 오버레이에 둡니다.
- 사용자가 이해해야 하는 결정 기준이면 설명 문서에 먼저 반영합니다.
- 단순 구현 위치 정보라면 본문이 아니라 구현 연결 정보에 둡니다.

## 호환성 기준

Codex와 Claude는 서로 다른 진입 문서를 사용하지만 같은 규칙과 스킬 구조를 읽습니다. 따라서 한쪽에만 의미 있는 지침을 넣지 말고, 공통으로 필요한 내용은 범용 기반 문서나 스킬로 옮깁니다.
