# 보고와 질문 스타일

## 한 문장 정의

에이전트가 사용자에게 보고하거나 질문할 때 정확함을 최우선으로 두되, 사용자가 이해할 수 있는 평어로 전달하는 규칙입니다.

## 핵심 원칙

길이가 아니라 단어 난도가 문제입니다. 사용자가 한 번에 이해할 수 있는 말이면 길어도 좋고 질문이 여러 개여도 좋습니다. 정확함을 깎으면서까지 쉽게 풀거나 짧게 줄이지 않습니다.

우선순위: 정확 > 이해 가능 > 짧음.

## 청중 가정

이 프로젝트의 기본 사용자는 시스템 owner이고, AI/소프트웨어 일반 용어에는 일정 수준 익숙합니다. 초보자에게 설명하듯 모든 단어를 풀어쓰지 않습니다. 단 사용자가 모를 가능성이 있는 다음 범주는 풀이가 필요합니다.

- 이 프로젝트 내부 신조어. 예: skeleton, overlay, ownership lock.
- 잘 알려지지 않은 영어 약어.
- 추상적인 영어 명사구. 예: mutation surface, write policy.
- 한자어 중 의미가 흐려질 수 있는 것. 예: 위임, 흡수, 승격.

## 풀이 방식

- 일반 영어 기술 용어를 한국어 문장에 그대로 박지 않습니다. 꼭 써야 하면 첫 등장 시 괄호로 한 줄 풀이를 붙입니다. 예: "overlay(프로젝트가 스켈레톤 기본값 위에 덮어쓰는 설정)".
- 같은 대화 안에서 같은 용어를 매번 풀어쓰지 않습니다. 첫 등장 한 번만 풀고 이후는 원문 그대로 사용합니다.
- 풀이가 원문보다 부정확해질 것 같으면 풀이를 빼고 원문만 둡니다. 정확도가 깎이는 풀이는 도움이 되지 않습니다.

## 원문 유지 범주

다음 범주는 풀어쓰지 않습니다. 풀어쓰면 의미가 흐려지거나 표준에서 벗어납니다.

- 보안 용어: privilege escalation, race condition, sandbox escape, buffer overflow, replay attack, SSRF, XSS, CSRF, MITM 등.
- 표준 프로토콜과 암호학: TLS, OAuth, JWT, SAML, HMAC, AES 등.
- 표준 컴퓨터과학 용어: idempotent, monotonic, atomic, transaction 등. 단 사용자가 모를 것 같으면 첫 등장 시 한 번만 괄호 풀이가 가능합니다.
- 코드, 명령어, 파일 경로, 설정 키 이름. 옆에 붙이는 설명만 쉽게 씁니다.

## 형식

- 보고가 길어도 괜찮습니다. 짧게 줄이려고 사실을 빼지 않습니다.
- 질문은 여러 개여도 괜찮습니다. 다지선다 옵션 라벨도 같은 평어 규칙을 따릅니다.
- 헤더, 리스트, 표는 내용에 갈래가 있을 때 적극 사용합니다. 단 한 단락 분량의 평문 응답에는 헤더를 붙이지 않습니다.

## 적용 범위

- 채팅 응답, 보고서 요약, 질문, 옵션 라벨, 다음 액션 제안.
- 코드 안의 식별자와 주석은 기존 코드 스타일 규칙(`rules/common/code-style.md`)을 따르고 이 규칙은 강제하지 않습니다.
- 사용자가 명시적으로 다른 언어를 요청한 경우 그 언어의 일반 규칙을 따릅니다.

## 결과는 무엇인가

이 규칙을 따르면 다음 상태를 유지합니다.

- 사용자가 보고를 한 번에 이해할 수 있습니다.
- 보안과 표준 도메인 용어의 의미가 흐려지지 않습니다.
- 같은 용어를 매번 풀어쓰지 않아 보고가 불필요하게 길어지지 않습니다.
- 정확함을 깎으면서 형식을 만족시키는 일이 없습니다.

## 기능 추가/수정 판단 기준

새 항목을 추가할 때는 다음을 확인합니다.

- 모든 프로젝트에서 동일하게 적용 가능한 기준인지.
- 기존 항목과 충돌하지 않는지.
- 정확함을 깎지 않는 항목인지.

특정 프로젝트에만 해당하는 톤이나 약속은 그 프로젝트의 `docs/PROJECT_PROFILE.md`나 운영 계획에 둡니다.
