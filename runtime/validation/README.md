# 검증 결과 기록

## 한 문장 정의

이 영역은 프로젝트의 검증 계획, 원시 결과, 요약, 평가자 메모를 남기는 곳입니다.

## 무엇을 하는가

검증은 구현자가 성공을 주장하기 전에 외부 관점에서 실제로 성공 기준을 만족하는지 확인합니다. 이 영역에는 마이크로, 미디엄, 풀 검증 결과와 실패 유형을 기록합니다.

## 왜 필요한가

AI 작업은 실행 결과를 낙관적으로 해석하기 쉽습니다. 검증 결과를 구조화해 남기면 성공, 부분 진전, 환경 문제, 정책 차단을 구분할 수 있고 다음 결정도 더 정확해집니다.

## 어떻게 동작하는가

검증은 세 단계로 나눕니다.

- Micro: 가장 싼 단일 개념 검증입니다.
- Medium: 한 하위 시스템이나 주요 단계를 검증합니다.
- Full: 사용자 관점의 end-to-end 검증입니다.

결과는 다음 값으로 분류합니다.

- `genuine_success`: 의도한 경로로 목표에 도달했습니다.
- `partial_progress`: 의미 있는 진전은 있지만 완전 성공은 아닙니다.
- `timeout`: 평가 시간이 초과됐습니다.
- `infra_error`: 프로젝트 품질과 무관한 환경 또는 도구 문제입니다.
- `blocked_by_policy`: 규칙이나 정책 때문에 진행하지 못했습니다.

## 결과 기록 템플릿

```json
{
  "project": "<project-name>",
  "phase": "micro",
  "trial": 1,
  "outcome": "partial_progress",
  "duration_seconds": 0,
  "effectiveness_score": null,
  "evidence": [],
  "notes": ""
}
```

## 결과는 무엇인가

검증 기록을 남기면 다음을 판단할 수 있습니다.

- 실제 성공인지 부분 진전인지
- 실패가 구현 품질 문제인지 환경 문제인지
- 다음 반복에서 무엇을 바꿔야 하는지
- 피벗 기준에 도달했는지

## 기능 추가/수정 판단 기준

검증 형식을 바꿀 때는 결과 분류와 피벗 판단에 도움이 되는지 확인합니다. 품질 지표에서는 `infra_error`와 `blocked_by_policy`를 제외하지만, 감사 기록에는 남깁니다. LLM 비결정성이 결과에 영향을 주면 여러 번 검증합니다.
